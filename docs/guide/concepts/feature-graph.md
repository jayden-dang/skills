# The feature graph

`check-trace` is the **vertical** layer: it proves that one feature's requirements, tasks, tests, and commits agree with each other.

`check-graph` is the **horizontal** layer: it answers a different question — *which features touch the same code, and does this new idea already exist somewhere?*

Both read the same specs. Neither requires anyone to author anything new.

## The problem

An agent asked to "add a keyboard shortcut for switching modules" has no way to know that a feature called `CHIPUI` already owns half the files it is about to edit, and that its Out-of-Scope section explicitly declined keyboard shortcuts three months ago.

The information exists. It is spread across a dozen `design.md` and `tasks.md` files nobody is going to read.

## The vocabulary

From the repo's own `CONTEXT.md`:

> **Surface** — the set of source files a feature owns or touches, derived from its `design.md` and `tasks.md`.
>
> **Surface manifest** — one feature's Surface, split into `owns` and `touches`.
>
> **Reverse index** — the inverse mapping: a source path → the features that reference it.
>
> **Shared surface** — a path referenced by two or more features; the duplication signal.
>
> **Summary card** — a bounded per-feature digest (code, name, owned paths, interfaces, out-of-scope) used to load a neighbor's essence without its full spec.

## How it works

`check-graph --harvest` walks `docs/specs/*/`, parses each feature's `design.md` and `tasks.md` for file paths, classifies each path as `owns` (it appears under a Create block) or `touches` (it appears under Modify), and renders `docs/specs/GRAPH.md`.

Crucially it **authors nothing new**. The source of truth is the specs that already exist. A path already written in a task's `Files: Create:` block is the manifest.

The generated file has two parts — the cards and the shared-surface table:

```markdown
# Feature Graph

<!-- GENERATED by `check-graph --harvest`. Do not edit by hand; regenerated by sync-spec. -->

## Cards

### FGRAPH — Horizontal feature-graph layer
- owns: scripts/check-graph.mjs, scripts/vendor-linters.mjs, …
- touches: scripts/check-trace.mjs, scripts/check-trace.test.mjs
- interfaces: harvest(specsDir, cfg) → Graph, query(graph, {paths, keywords}) → …
- out-of-scope: Manually-declared typed edges … | Semantic or embedding-based matching …

## Shared surface

| Path | Features |
|---|---|
| scripts/check-trace.mjs | FGRAPH:touches, TRACE:touches |
```

## The three modes

| Mode | What it does |
|---|---|
| `--harvest` | Writes `<specsDir>/GRAPH.md`. Run by [`sync-spec`](../skills/sync-spec.md); the result is staged into the same commit |
| `--query --json [--path P]… [--keyword K]…` | Returns the features overlapping those paths or keywords, with their summary cards |
| `--verify` | Lints two things: the committed `GRAPH.md` is byte-identical to a fresh render, and every harvested feature code is registered in `INDEX.md` |

`--verify` failing on staleness is the mechanism that keeps the committed graph honest. It is wired into pre-push and CI by `setup-repo`, and [`verify`](../skills/verify.md) requires it to pass before the claim "requirements met" may be spoken.

## Where it is consumed

Exactly two gates read the graph, and both do so *advisorily*.

**[`brainstorm`](../skills/brainstorm.md), step 1.** Before the interview begins, it runs the dedup check with the scan's candidate files as `--path` and the idea's key terms as `--keyword`. Any returned features are presented as their **summary cards, not their full specs** — each card's owned paths and Out-of-Scope list show what the neighbor already covers.

Its completion criterion is a sentence the agent must be able to say: *which existing features share this idea's surface and how the new idea differs, citing feature codes — or that no existing feature shares its surface.*

**[`code-review`](../skills/code-review.md), step 3a.** It queries with the diff's changed source files. When overlaps come back, the **Spec** subagent additionally receives the neighbor cards, with a brief directing it to flag — as a *reuse-miss* finding citing the neighbor's feature code — any place the diff reimplements behavior a shares-surface neighbor already owns.

## Graceful degradation is a designed behavior

A repo that never installed the graph must not be nagged about it on every turn. Both consuming skills carry the same three-way handling, and it is worth quoting because it is easy to get wrong:

- **`check-graph` is absent** → the feature graph is not installed in this repo. Say so, name `setup-repo` as the remedy, and say it **at most once per session**.
- **`check-graph` errors** → note that the automated overlap check was unavailable.
- **Either way** → continue with manual exploration, and **never block the gate on it**.

The graph is an advisory signal. It never fails a review and it never stops a brainstorm.

## Vendoring and drift

The linters live in the skill set's own `scripts/` — a single canonical copy, deliberately not mirrored into `templates/`, so there is exactly one axis of drift to police. But the *consuming repo* runs them, which means they must be copied in.

`scripts/vendor-linters.mjs` does that, and stamps each installed file with a hash of its own body:

```js
// @skills-linter: check-graph sha256:8897837c7133
```

`setup-repo` calls it with `--install` on first setup and with `--check` on re-runs. The check reports each linter as one of four states:

| State | Meaning |
|---|---|
| `ok` | current |
| `missing` | never installed |
| `outdated` | the skill set moved on |
| `modified` | someone edited the repo's copy |

On any drift, `setup-repo` reports which linters drifted and **offers** to update. It never overwrites a `modified` copy without the user's explicit yes — that copy may carry a local fix worth upstreaming.

## What it deliberately is not

The `FGRAPH` spec's Out-of-Scope section is unusually informative, because several of its entries record *evaluated and declined* ideas rather than merely unbuilt ones:

- **Typed edges** (`extends`, `depends-on`, `supersedes`) and their automatic reversal. v1 derives only the untyped `shares-surface` relationship, which is free from the harvest.
- **Authoritative owns-vs-touches precision.** Role classification is best-effort; the dedup signal depends only on shared-surface *presence*.
- **Semantic or embedding-based matching.** Keyword matching only.
- **Harvesting surface from committed code or import graphs.** The source of truth is specs only — a spike proved specs carry enough signal.
- **Card-based context loading inside `write-design` / `execute-plan`.** Evaluated and *declined*: a throwaway A/B prototype on a real cross-feature task showed two cold planners made identical reuse decisions with and without the card, and the card's `interfaces` field omitted the interfaces the task actually needed while including irrelevant ones. `write-design` already bounds context via its scan-subagent digest, and `execute-plan`'s task briefs already carry exact per-task neighbor `file:line` references. Neighbor-awareness is delivered at the two gates that matter.

That last entry is the system's own methodology applied to itself: a design question got a runnable answer via [`prototype`](../skills/prototype.md), the answer was negative, and the answer — not the prototype — was captured durably.

## Known limits

The `FGRAPH` spec records a measured scale problem, honestly. On a 4-feature repo, `GRAPH.md` renders at 15.7 KB, with per-card Out-of-Scope prose consuming 35–53% of card bytes. A 52-feature simulation renders 140.6 KB (roughly 36k tokens); one hub path returns 38 full cards at 97.3 KB; and the raw-overlap ranking degenerates to alphabetical order when every candidate overlaps on the same hub file.

Fixing this supersedes several approved criteria and has live design alternatives, so it escalated to `brainstorm` rather than being decided inside an amendment. Which is, again, the system behaving as designed: the "small change" grew a design decision with real alternatives, so [`amend`](../skills/amend.md) handed it up.

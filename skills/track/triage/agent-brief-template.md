# Agent briefs

An agent brief is the comment posted on an issue when it moves to `ready-for-agent`. It is the specification a future agent will implement from, possibly weeks later, with no access to today's conversation. The brief is the contract; the issue body and thread are background.

For a PR, the brief describes what remains to be done **to the existing diff** — finish it, close its gaps, address review points — rather than building from nothing. Same fields, same rules.

## Rules

**Durable over precise.** The codebase will change while the issue waits. Name behavioral contracts, types, function signatures, and config shapes — things an agent can grep for after a refactor. Never cite file paths or line numbers; they go stale and mislead.

**Behavioral, not procedural.** Say what the system must do, not which edits to make. The implementing agent explores the code fresh and makes its own structural choices.

**Every acceptance criterion independently verifiable.** Each checkbox must be checkable on its own with a concrete command, test, or observation. "Works correctly" is not a criterion. When the issue touches behavior covered by a spec, cite the requirement IDs (`CODE-N.M`) in the relevant criteria so the trace stays intact — and add criteria for any guard requirements the change must not break.

**Scope is explicit.** The Out of scope list is what stops an agent from gold-plating or wandering into adjacent features.

## Template

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** bug | enhancement
**Summary:** <one line: what must happen>

**Current behavior:**
<What happens today. For a bug: the broken behavior, with reproduction
conditions. For an enhancement: the status quo being built on. For a PR:
the state of the diff — what works, what is missing.>

**Desired behavior:**
<What must happen after the work is done. Cover edge cases and error
conditions explicitly.>

**Key interfaces:**
- <`TypeName` — what about it changes and why>
- <`functionName()` contract — current vs required behavior>
- <config/data shape — new fields and their meaning>

**Acceptance criteria:**
- [ ] <independently verifiable criterion — cite CODE-N.M where spec'd>
- [ ] <...>

**Out of scope:**
- <adjacent thing that must NOT be touched>
```

## Calibration example

A report says CSV export writes `NaN` for missing numeric cells.

Weak brief: "Export is broken for empty cells — see the export handler around line 200 in src/export/csv.ts and fix it." Everything wrong at once: no category, a file path and line number that will rot, no definition of correct output, nothing checkable.

Strong brief (abridged): Category `bug`; current behavior "numeric columns serialize missing values as the string `NaN`"; desired behavior "missing values serialize as an empty field, regardless of column type"; key interface "the value-serialization contract used by the CSV writer — whatever function maps a cell value to its field text"; criteria "a row with a missing numeric cell produces `a,,c`", "existing populated-cell output is byte-identical (guard — cite the export feature's guard ID if one exists)"; out of scope "changing the delimiter or quoting rules".
